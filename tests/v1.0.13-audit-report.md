# AgentScribe v1.0.13 — Regression Audit Report

Date: 2026-05-20
Auditor: MAX (regression sweep against waves 1-6 + 3 hotfixes)
User report: "Sessions page is broken — no data showing."

---

## Verdict

The blank sessions page is **not caused by broken imports, renamed handlers, missing DOM IDs, or schema mismatches**. Every contract the renderer depends on still resolves cleanly. The sessions page CAN render — but there is one regression in the **stop-popup race fallback** that reliably presents the user with a blank sessions tab right after they stop recording on the overlay.

### Smoking gun — Stop-from-overlay opens sessions.html BEFORE the session is persisted

`background.js:865-895` (`STOP_RECORDING_FROM_OVERLAY` handler) does, in order:

1. Fires `chrome.action.openPopup()` (often rejected from SW context without a popup-side user gesture).
2. After **300 ms**, if the popup didn't open, opens `sessions/sessions.html` in a new tab.
3. **Then** kicks off `stopRecording()` asynchronously — CDP detach + correlation + name inference + `chrome.storage.local.set({ completedSessions })`. On any non-trivial session this takes 1-3 s; on heavy pages 10-60 s.

When sessions.html loads inside that 300 ms-to-stopRecording-complete window:

- `sessions.js init()` immediately calls `GET_COMPLETED_SESSIONS`.
- Background returns the **pre-stop** `completedSessions` array. The just-finished session is not in it yet.
- `sessions.js` has **no `chrome.storage.onChanged` listener**, so when `stopRecording()` finally writes the new session, the rendered page never updates.
- Result: if this is the user's first recording, the page renders the empty state forever. If they've recorded before, the new session is silently missing until they hit refresh.

This matches "no data showing" verbatim when the user is testing a single fresh recording.

### Why I'm confident this is the regression vs. pre-existing

The race exists because Wave 6's stop-popup fix added the 300 ms tab-open fallback. v1.0.12 stopped exclusively via the popup, which means the popup itself was loaded after stopRecording completed (or at least concurrently, populated by GET_LAST_SESSION on its own poll). The fallback path is new in v1.0.13 and was not paired with an auto-refresh on the sessions side.

---

## Fixes applied in this audit

### 1. sessions.js — add `chrome.storage.onChanged` auto-refresh (HIGH)

Single-file edit, additive. When `completedSessions` changes in storage, the sessions page re-renders. This closes the race regardless of how slow `stopRecording()` is and regardless of whether the popup fallback fires.

Comment block added inline noting why.

## Things that are NOT bugs today but worth watching

1. **`chrome.action.openPopup()` reliability**: per Chrome MV3 docs, this requires a popup-side user gesture. The current SW-context call relies on transient gesture propagation from `chrome.runtime.sendMessage` from the overlay. This works in modern Chrome but can break with browser updates. The 300 ms fallback is the safety net — but as documented above, the fallback creates the blank-page race that this audit fixes. With the new `storage.onChanged` listener, the race is harmless: the fallback can open early and the page will fill in when stop completes.

2. **Storage quota on heavy sessions**: a single session with full `bundleFindings`, `storageSnapshots`, `wsFrames`, `fetchEvents`, and 1000+ network events can be ~5-15 MB. With `unlimitedStorage` permission this is fine, but 20-session retention can push toward 100+ MB. No telemetry warns the user. Not a regression — pre-existing capacity issue.

3. **`STOP_RECORDING_FROM_OVERLAY` does not handle a stoppedRecording promise rejection**: if `stopRecording()` throws, `sendResponse(result)` sends `undefined`. The overlay sender has no rejection handler. The user just sees no popup open. Not the current symptom but worth a follow-up.

4. **`fetchEvents` is not surfaced in any exporter except possibly bundle/mcp**: Wave 6 added the field, but the exporters were finalized in Wave 4 before the field existed. Not a runtime bug — data is captured but partially un-exported. Follow-up: extend mcp/bundle exporters to include `fetchEvents` alongside `networkEvents` for replay completeness.

5. **`sessions.js` `sendMessage()` returns `null` on `chrome.runtime.lastError`** — the sessions page silently swallows the error and shows an empty state. If the SW crashes on boot, the user can't tell whether they have zero sessions or the SW is dead. Minor UX bug — not the cause of the report.

## Recommendation for the user

After applying the fix:

1. Reload the unpacked extension at `chrome://extensions`.
2. If your `completedSessions` is currently empty (e.g. because the v1.0.13 race lost a recent stop), record one fresh workflow end-to-end. Stop via the overlay. Confirm the sessions page now auto-populates within ~1-3 seconds without manual refresh.
3. If you still see no data, open the sessions page DevTools console (F12 from the sessions tab) and screenshot any red error. With the verified-clean import graph, a red console error from sessions.js would point to a deployment issue (cached old build, stray browser-cache, etc.) rather than a code regression.
