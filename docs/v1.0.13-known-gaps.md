# AgentScribe v1.0.13 — Known Gaps

**Authored by**: `wave5-adversarial`
**Scope**: honest gap inventory after Wave 1–4 ship. No hand-waving.
**Method**: code review only — real-world Cloudflare/Etsy replay was out of reach. Every claim below cites a file + function.

The headline: Waves 1–4 close **most** capture gaps, but several detectors that were forged in Wave 3 are **declared but never invoked**, and any challenge layer harder than "the cookie was present at recording time" will re-trigger on replay. The exporters silently assume capture happened. They don't crash when it didn't, but the agent will think the artifact is complete when it isn't.

## 2. Anti-bot challenge re-trigger

The implementation **detects** anti-bot layers via `field-scanner.js#detectChallengeLayer` (cookie sniffing for `cf_clearance`, `_pxhd`, `_abck`; iframe/script src matching for Turnstile, PerimeterX, Akamai, hCaptcha, reCAPTCHA). But:

1. **The detector is never invoked.** No file calls `window.__agentscribe.detectChallengeLayer(document)`. `session.challengeLayer` is `undefined` in every recorded session. The exporters' warning comments are therefore dead-code-gated — they will never fire in practice until a Wave 5.1 patch wires `content.js#scanFields` (or a sibling) to call it and ship via `chrome.runtime.sendMessage`.

2. **Even if the detector fires, replay will fail per provider:**

### Cloudflare Turnstile / `cf_clearance`
The `cf_clearance` cookie is bound to (a) client TLS fingerprint, (b) source IP, (c) user-agent string, and (d) browser JS challenge result. Injecting the cookie into a fresh `chromium.launch()` context will fail at the next CF challenge check because:
- Bare Playwright Chromium has a different TLS fingerprint (no Chrome-specific cipher order, no JA3 match).
- The CF challenge JS re-runs on most page loads and validates the cookie's origin.
- The IP is often re-checked.

**Recommended mitigations** (none implemented, all out of scope for v1.0.13):
- `playwright-extra` + `puppeteer-extra-plugin-stealth` (overrides webdriver flag, navigator props, WebGL fingerprint). Playwright exporter mentions this in a comment (`playwright-exporter.js:14-19`) but does not generate any code change.
- Drive a real Chrome session via `chrome.debugger` (CDP) attached to the user's existing browser. AgentScribe could in theory ship a "replay" mode that re-uses the recording session's CDP attach — out of scope.
- Run replay through the user's IP via a residential proxy (legal/operational concern — out of scope).

### PerimeterX (`_pxhd`, `_px3`, `_pxvid`)
Same story plus a stricter behavioral model — captures mouse paths, keystroke timing, scroll velocity. Cookie injection alone leaves the `_px3` server-side state unsynchronized. Replay re-triggers a challenge gate. No mitigation in v1.0.13.

### Akamai Bot Manager (`_abck`, `bm_sz`, `ak_bmsc`)
Akamai sensor data is collected continuously by the loaded `akam-` script. The sensor payload is POSTed back. Replaying without re-running the sensor script (or with a stale `_abck` cookie) → "validation expired" → block. Recommendation: capture the sensor POST endpoint and the script URL during recording; agent must arrange to re-run the sensor script on replay.

### hCaptcha / reCAPTCHA
Token-based, single-use, IP-bound, time-bound. Cookie/storage injection is useless. Tokens are NOT recoverable from a recorded session. Replay MUST either (a) pause and ask the user to solve the challenge live, or (b) use a paid solving service (legal/ethical line — not recommended). `playwright-exporter.js` has no breakpoint emission for captcha events; only OTP breakpoints exist (`eventNeedsOtpBreakpoint`).

**Bottom line for the agent**: if `bundleFindings` or any URL contains `challenges.cloudflare.com`, `perimeterx.net`, `akam-`, `hcaptcha.com`, or `google.com/recaptcha`, **flag the session as requiring stealth-browser replay** and DO NOT silently fall through to `playwright` exporter expecting a clean run.

## 3. OAuth multi-tab flows

`background.js:408-426` handles `chrome.tabs.onCreated`. The check is:

```js
if (tab.openerTabId && trackedTabIds.has(tab.openerTabId)) {
  trackedTabIds.add(tab.id);
  // ... attach CDP, inject scripts, push to tabsAttached[]
}
```

Behavior per OAuth pattern:

### Google OAuth (popup → consent → close popup → parent receives token via `postMessage`)
**CAPTURED.** The popup is opened via `window.open(...)`, which sets `openerTabId`. The popup joins the session. CDP attaches. Cookies on `accounts.google.com` are snapshotted. The `postMessage` payload back to the parent **is NOT** captured — content.js doesn't hook `window.addEventListener('message')` for generic messages (only the `agentscribe-ws` envelope). The agent will see network events on both tabs but may miss the actual access_token if it's only delivered via `postMessage`. Agent fallback: inspect `tabsAttached[]` and correlate the OAuth callback URL with the parent's subsequent API requests.

### GitHub OAuth (same popup pattern)
**CAPTURED**, same caveat. GitHub typically returns `code=...` in a redirect URL the popup navigates to, which `webNavigation.onCommitted` will see — so the code is in the network log. The token exchange is `POST /login/oauth/access_token` from the parent app, which is captured.

### `window.open(url, 'named_window')` reusing a named tab
**PARTIAL.** First call creates a tab with `openerTabId` — captured. Subsequent calls reusing the name navigate the same tab — captured by `webNavigation.onCommitted` on that tab. No issue.

### Top-level redirect (no popup) — e.g. SAML SSO, some Auth0 flows
**CAPTURED via webNavigation.** No popup, no `tabs.onCreated`, but the user's existing tab navigates to the IdP. `webNavigation.onCommitted` fires (`background.js:453-481`), cookies are snapshotted on each navigation, and the round-trip is recorded. The IdP's cookies will be in `cookieSnapshots[]` with the IdP's domain.

### Redirect through `about:blank` intermediary
**MISSED partially.** Some OAuth providers use a transient `about:blank` window. `webRequest`/CDP don't capture much on `about:blank`. If the IdP's auth response lands in `about:blank` briefly before redirecting, the auth code may be visible only in URL — depending on timing, may be lost.

### Cross-window `BroadcastChannel` / shared `localStorage` event
**NOT CAPTURED.** Some SPAs sync auth between tabs via `BroadcastChannel` or `storage` events. Neither is hooked. The token's arrival on the parent tab would still appear in the `localStorage` snapshot at the next `runStorageSnapshot('pagehide')`, but the timing/causality is lost.

**Agent verification step**: for any session with `tabsAttached.length > 1`, manually review the `cookieSnapshots[]` and `storageSnapshots[]` per origin to verify the token landed on the expected origin.

## 4. HMAC-signed requests

`auth-detector.js` detects HMAC presence (header regex `HMAC_HEADER_RE = /^(x-signature|x-hmac|x-hmac-signature|x-auth-hash|x-amz-signature)$/i`) or `signature=|hmac` in the Authorization header value (`auth-detector.js:270-276`). Sets `auth_scheme: 'hmac-signed'` with confidence 0.75–0.85.

`bundle-analyzer.js#scanSigningFunctions:102-128` static-scans for:
- `CryptoJS.HmacSHA\d+` → `HMAC-SHA{N}`
- `createHmac('sha\d+'|'md5', ...)` → `HMAC-{ALGO}`
- `crypto.subtle.sign(` → `'custom'`
- `crypto.subtle.importKey(...HMAC...)` → `'HMAC-custom'`

### What IS detected
- Unminified CryptoJS-using SDKs (rare in modern bundles).
- Server-side-style Node `createHmac` that got browser-bundled.
- WebCrypto subtle.sign with HMAC importKey nearby.

### What is NOT detected
- **Minified/mangled bundles** where `CryptoJS.HmacSHA256` got rewritten to something like `r.HSh(a, b)`. The regex matches the literal string, not semantics.
- **Custom HMAC implementations** written in pure JS (loops over bytes, XOR pads, manual SHA). No regex catches them.
- **WASM-based signers** (Solana, some crypto APIs) where the signing function is a `WebAssembly.instantiate` import. Static analysis sees the WASM URL fetch but cannot decompile.
- **Signers loaded dynamically** via `import()` from a code-split chunk that didn't fire during the recording (the chunk URL is in the manifest but its source isn't fetched by `content.js#runBundleAnalysis` because the regex scans only `script[src]` tags present at `load`).
- **Signers running in a Web Worker** — `content.js` does not inject into Workers, so the Worker's source is invisible.
- **Signers in Service Workers** — same.
- **String-builder obfuscation** — `'Hm' + 'acSHA' + '256'` defeats the regex.

### What the agent must do when `auth_scheme: 'hmac-signed'` but `signing_functions: []`

Inspect `bundleFindings[*].findings.signing_functions` — if empty:

1. **Manual DevTools pass**: open the live target, click through to the action that triggers the signed request, set a Network breakpoint on the suspect endpoint, then use the call stack to walk back to the signer.
2. **Identify the key source**: HMAC requires a key. It's typically in localStorage (some auth `client_secret`-style value), in a cookie, or derived from a fetched config. Check `storageSnapshots[*].snapshot.localStorage` for candidate keys.
3. **Identify the signed payload**: log the request body + headers + URL canonicalization (AWS SigV4 style, or simpler `${method}\n${path}\n${body}`).
4. **Port to replay**: write the signer in Node/Python and call it from the Playwright `route()` interceptor before each request. AgentScribe will NOT generate this for you.

This is the single hardest manual step that survives v1.0.13. Sites with custom HMAC will require ~20-60 minutes of human reverse-engineering per integration. No way around it without active deobfuscation tooling, which is out of scope.

## 5. Service Worker fetch interception

WAVE_PLAN_v1.0.13.md row 21 promised "Page-context fetch/XHR proxy via injected content script." **This is not implemented.**

Search the repo: no `window.fetch =` wrapper, no `XMLHttpRequest.prototype.open` patch. `ws-capture.js` ONLY wraps WebSocket + EventSource. Confirmed by Grep over the source tree.

### When this matters
- **Gmail PWA** — all mail fetches go through `/sw.js`. `chrome.webRequest` sees the requests the SW MAKES (network-side), so most Gmail traffic IS captured because the SW does eventually hit the network. But requests served from the SW cache are invisible — they never hit `webRequest`.
- **Twitter / X PWA** — similar SW routing.
- **YouTube PWA** — heavy SW caching.
- **Notion / Linear** — extensive SW use.
- **Discord** — heavy SW + WebSocket routing.

### What works today
- `chrome.debugger` (CDP) `Network.requestWillBeSent` events fire for requests the SW makes to the network. These are captured. ✓
- The page's `fetch()` calls that the SW intercepts and proxies to network — captured. ✓
- The page's `fetch()` calls that the SW serves from CacheStorage — **invisible**. The fetch never hits the network layer. Neither CDP nor `webRequest` sees it. ✗

### Workaround for the agent
Open DevTools → Network panel during the recording. If you see requests in DevTools that DON'T appear in `session.networkEvents`, those were SW-cache-served. Mark the session as "incomplete network capture, SW-cached responses missed."

### What v1.0.14 should add
Inject a MAIN-world script (sibling of `ws-capture.js`) that:
```js
const _fetch = window.fetch;
window.fetch = async (...args) => {
  const start = Date.now();
  const resp = await _fetch(...args);
  const clone = resp.clone();
  window.postMessage({source:'agentscribe-fetch', request:{url, headers, body}, response:{status, headers, body: await clone.text()}, timing:{start, end:Date.now()}}, '*');
  return resp;
};
```
Plus an `XMLHttpRequest.prototype.open/send` patch for legacy XHR. Plus a `navigator.serviceWorker.controller` check to flag SW-controlled origins so the agent knows to verify capture completeness.

## 6. Mobile / iframe / cross-origin worker capture

`manifest.json:29` sets `"all_frames": true`, so `content.js` runs in every frame.

### Same-origin iframes
**CAPTURED.** Each frame gets its own `content.js` instance, fires its own `STORAGE_SNAPSHOT`, `DOM_EVENT`, etc. Background dedupes by `sender.tab?.id` but not by frameId — events from a same-origin iframe are tagged with the parent tab's id and merged into `events[]`. URL on each event records which page the event came from.

### Cross-origin iframes (e.g. embedded Stripe checkout, embedded YouTube)
**PARTIAL.**
- `content.js` runs in the cross-origin frame (per `all_frames:true`).
- `storage-snapshot.js#dumpWebStorage` reads `localStorage` for THAT origin (not parent's). So a Stripe iframe's `localStorage` IS captured — but tagged under the parent tab's session, with the URL field being the iframe's URL.
- `chrome.cookies.getAll({url: parentUrl})` does NOT capture the iframe's cookies — different origin. Cookies for the cross-origin iframe's origin are simply absent from `cookieSnapshots[]`. **This is a real gap.** Stripe iframe sessions, embedded payment forms, embedded SSO — their cookies won't be in the bundle.
- Mitigation: a v1.0.14 patch should iterate `webNavigation.getAllFrames` per tab and call `chrome.cookies.getAll` per frame URL.

### Sandboxed iframes (`sandbox=""` without `allow-scripts`)
Content scripts can't run. Capture is impossible. Affected sites: some bank iframes, some payment processor iframes by design.

### Mobile Chrome
AgentScribe is a desktop Chrome extension. Mobile Chrome doesn't support extensions in stable. Out of scope. No code path for mobile capture.

### Web Workers / Service Workers
Not captured. Content script can't run in workers. Any state held only in a worker (some crypto libs do this) is invisible.

## 7. Privacy considerations

The exported bundle JSON contains, in `auth_state` (mcp-exporter) and `storageState` (playwright-exporter):

- **Full session cookies** including `httpOnly` and `secure` flagged ones — `chrome.cookies.getAll` ignores both flags, exposes everything to the extension. This is the same access F12 has and the same shape Playwright needs.
- **All localStorage entries** — typically includes JWT tokens, refresh tokens, user IDs, in-app secrets.
- **All sessionStorage entries** — often holds CSRF tokens, transient auth state.
- **JWT decoded payload** — surfaces `sub` (user identifier), `email`, `iss`, custom claims.
- **CSRF token values** — captured as raw `value` on hidden CSRF inputs (`field-scanner.js:303`).
- **Form input values** — DOM input events, redacted ONLY if a field matches `SENSITIVE_INDICATORS` (password/credit/cvv/ssn/secret/token); everything else is logged verbatim. Email addresses, names, dollar amounts, internal IDs — all captured raw.
- **Request and response bodies** — full POST bodies, full JSON response bodies (`responseBodyParsed` in `background.js`).
- **WebSocket payloads** — every frame, base64 if binary.

### Where the file lands
`background.js#handleExport` returns content + filename. Where it's WRITTEN depends on the popup's invocation of `chrome.downloads.download`. By Chrome default, downloads go to `~/Downloads/`. There's no enforced subdirectory `Downloads/AgentScribe/sessions/` — the manifest declares `downloads` permission (`manifest.json:15`) but the export path isn't enforced subdir. **Verify with the user where their browser saves downloads.**

### Operational rules for the bundle
1. **The bundle JSON is a full credential artifact.** Sharing it = sharing the user's authenticated session for every captured origin until those cookies/tokens expire.
2. **Never commit a bundle JSON to a public repo.** Add `*.bundle.json`, `agentscribe-*.json`, and `agentscribe-playwright-*.js` to `.gitignore` in any project that consumes them. The playwright-exporter inlines `storageState` directly into the script — that file IS the credential too.
3. **Never paste raw bundle into untrusted agents/services.** A third-party LLM API call with the bundle in the prompt = credential leak to that vendor.
4. **Treat the bundle like a session cookie file from a stolen laptop.** Rotate the affected logins if the bundle leaves your control.
5. **Time-bind use.** Most session cookies expire in hours-to-days. JWTs in localStorage usually 15min-1hr. Don't keep bundles around past replay; delete after use.
6. **No `logSensitiveValues` setting.** The settings UI has it (`background.js#getSettings` default `false`) and content.js respects it for input event values — but localStorage / cookies / response bodies are captured regardless. The setting only redacts DOM-event input values for fields whose attrs match `SENSITIVE_INDICATORS`. It does NOT broadly redact authentication material.

### Recommended v1.0.14 hardening
- Encrypt bundle JSON at rest with a user-supplied passphrase via WebCrypto AES-GCM.
- Add an explicit "this bundle contains credentials" warning banner on export.
- Auto-delete `completedSessions` from `chrome.storage.local` after N days.
- Redact `Set-Cookie` and `Authorization` response headers from `networkEvents[*].responseHeaders` by default; gate on a setting to include them.

## 8. Recommended next iterations (v1.0.14 candidates)

In rough priority order based on this audit:

1. **Wire the dead detectors** (HIGHEST priority — these were forged but never invoked):
   - `field-scanner.js#detectChallengeLayer` → call from `content.js` on each `scanFields()` pass; ship via new `CHALLENGE_DETECTED` message → background sets `sessionBuffer.challengeLayer`.
   - `correlation-engine.js#inferPagination` → call from `background.js#finalizeNetworkEvent` after response body is available; assign to `event.pagination`.
   - `correlation-engine.js#tagMutation` → same path; assign to `event.mutates_state`.
   - Without these three wires, three exporter blocks (`challenge_layer`, `pagination_strategies`, semantic_endpoints' mutation override) silently emit empty/dead values.

2. **Page-context fetch/XHR proxy** (`window.fetch` + `XMLHttpRequest.prototype.open/send` wrappers in MAIN world, mirroring `ws-capture.js`). Closes row 21. Catches SW-cache-served requests by hooking the page side, not the network side.

3. **Full IndexedDB value snapshot** — currently keys + metadata only (`storage-snapshot.js#snapshotOneStore`). Add a `store.get()` per sampled key, bounded by total byte budget. Many SPAs store auth tokens / app state in IDB; without values the bundle is incomplete.

4. **Cross-origin iframe cookie capture** — iterate `webNavigation.getAllFrames`, call `chrome.cookies.getAll` per frame URL, merge into `cookieSnapshots[].cookies` with frame origin tagging.

5. **MCP-server wrapping AgentScribe** — flagged in WAVE_PLAN_v1.0.13.md line 200. Would let Pattern OS constructs (cpaswm, devswm) ingest sessions natively without manual file path passing.

6. **In-extension replay engine** — currently replay requires external Playwright. An in-extension replay using `chrome.debugger` to re-drive the SAME Chrome profile would bypass the TLS-fingerprint problem from §2 for free (the user's real Chrome IS the fingerprint).

7. **Anti-bot stealth integration** — wire `playwright-extra` + stealth into the generated Playwright export when `challengeLayer` is detected. Note: actual challenge bypass remains legally/ethically off-limits per CLAUDE.md and WAVE_PLAN out-of-scope. Stealth covers detection avoidance for sites the user is legitimately authorized to access; it does not solve captchas.

8. **Bundle encryption + retention policy** (§7).

9. **Redact captured form values that match common PII patterns** (email, SSN, credit card via Luhn) beyond just labeling sensitive fields. Currently every non-sensitive input value is logged verbatim.
